## Changes from Day 94

**Resolved this session:**
- Paper-trading launchd job was firing at 4:30pm ET while its own comment assumed 4:30pm CT (machine's actual local timezone is America/Toronto/Eastern, confirmed via `/etc/localtime`) — silently shrinking the intended ~90-minute post-close settling buffer to ~30 minutes. Fixed: schedule shifted to 17:30 ET, comment corrected, launchd reloaded. See Golden Rule 33 and `PROJECT_STATUS_DAY95_SHORT.md`.
- **Every data provider's circuit breaker miscounted ticker-specific data gaps as provider-health failures.** Found investigating why Tradier "didn't help" during a forward-test run: its breaker tripped from 3 failures that were actually 2 genuine ticker-coverage gaps (BRK.A/BRK.B — known cross-provider trouble tickers) plus one transient timeout, then blocked every other ticker in the run for a flat 5-minute cooldown even though a direct re-test proved Tradier was fully healthy (10/10 known-liquid symbols, all <0.5s). Root cause: every provider's `get_ohlcv`/`get_quote`/`get_fundamentals` methods called `breaker.record_failure()` immediately before raising `DataNotFoundError`/`InsufficientDataError` — the same failure count used to judge "is this provider down." A ticker having no data says nothing about the provider's health; conflating the two meant a couple of unlucky ticker misses could take a perfectly healthy provider out of rotation for everyone else mid-scan. **Fixed across all 6 providers** (Tradier: 5 sites, yfinance: 6, TwelveData: 1, Finnhub: 2, AlphaVantage: 1, Stooq: 3 — FMP was already correct) — `DataNotFoundError`/`InsufficientDataError` no longer count toward the breaker; genuine connectivity/timeout/rate-limit/auth failures still do. Verified with a 3-pass review (Golden Rule 32): (1) confirmed working via a live Tradier re-test, breaker correctly self-healed `OPEN→HALF_OPEN→CLOSED`; (2) checked no UI/endpoint reads `failure_count` or breaker state (none does, no downstream break); (3) confirmed no double-counting when a breaker is already open (`_check_availability()` blocks without recording another failure), and swept all 36 remaining `record_failure()` call sites across every provider to confirm each is still tied to a genuine health signal (429/401/5xx/timeout/connection error), not a data-coverage gap — a systemic provider-wide outage would still trip the breaker via those paths.
- **`.env` loading was fragile — only explicit in `backend.py`, everywhere else relied on import-order side effects.** `paper_trading/daily_job.py`, `live_signals.py`, and ad-hoc diagnostic scripts all got API keys loaded only because they happened to transitively import `backtest/simfin_loader.py` or `providers/backtest_adapter.py`, both of which call `load_dotenv()` themselves — a future refactor removing that import chain would silently break every provider's API key with zero warning. Fixed by loading `.env` explicitly once in `providers/__init__.py`, so every consumer of `from providers import get_data_provider` gets it regardless of import order. Verified: a bare `from providers.tradier_provider import TradierProvider` now loads its token automatically with no explicit `load_dotenv()` call needed.

**Added this session:** Momentum's R:R gate structurally caps at 0.80 for ~45% of otherwise-qualifying candidates (see Medium finding below) — found via a live TradingView-screener deep dive, not part of the earlier launchd-timezone fix.

**Freeze status:** unchanged — forward-testing accumulation remains the user's sole stated priority. Today's plist fix is scheduling/ops-only and does not touch the verdict engine or paper-trading gate.

### Medium: Momentum's R:R gate structurally caps out at 0.80 for ~45% of otherwise-qualifying candidates (new, found Day 95, not actioned)
**Severity:** Medium
**Description:** Deep-dived the TradingView screener pipeline and traced a live example (CXW) that passed Trend Template (8/8), RS vs SPY (1.37), fundamentals (Decent), and the full verdict engine (BUY) — then got rejected purely on R:R (0.80 < 1.2 minimum). Ran the same pipeline across 58 Config-C-qualifying candidates (mostly 7-8/8 Trend Template) to check how often this happens: **81.0% (47/58) failed the R:R gate**, and of those, **55.3% (26/47) hit the exact same 0.80** — not a coincidence, but a structural artifact of `compute_entry_levels()`'s "standard" holding period: target is fixed at entry×1.08 (+8%), and stop (`swing_low − 2×ATR`) is clamped between entry×0.90 and entry×0.97 (3-10%). Whenever a stock's natural ATR-based stop distance exceeds 10% — common for the higher-ADX, more-volatile names this scan is specifically selecting for — it clamps to the floor, making R:R mathematically fixed at 8/10 = 0.80 regardless of how strong the underlying setup is. Only candidates whose natural stop distance happens to land under ~6.7% (8/1.2) can ever clear the 1.2 minimum. Full measurement script: `/private/tmp/claude-501/.../scratchpad/trace_rr_clamp_stats.py` (scratchpad, not committed).
**Why it matters:** This is a strong candidate for the actual bottleneck behind momentum's slow signal rate (14 signals total since Day 81, 1 closed) — a bigger apparent effect than the known fundamentals-mismatch or candidate-pool-cap issues. It disproportionately filters out the *more volatile* qualifying setups in favor of calmer ones, which may or may not correlate with which trades would actually have won — that's unproven either way since rejected candidates generate no outcome data.
**Fix:** Not actioned — `compute_entry_levels()`'s target/stop formula is a frozen, pre-registered threshold (`PAPER_TRADING_PREREGISTRATION.md` §7). Per Golden Rule 18, it cannot be touched until the 100-trade paper-trading bar clears (any change resets the trade count to zero). Logged now so it's on record rather than re-discovered later; revisit alongside ROADMAP Priority #11 (volume-confirmation gap) once the freeze lifts as a candidate for the same design + re-backtest session.

**Decision (Day 95, same session — since superseded, see below):** User initially chose NOT to fix this now and pre-selected "widen the stop clamp floor to entry×0.85" as the approach for later. **This was corrected later the same session — that direction was wrong.** A quick backtest sanity check (baseline vs. floor=0.85 on the same 60-ticker/5yr universe) showed identical trade count/win-loss composition but *worse* PF (1.664→1.638), Sharpe (0.647→0.622), and drawdown (45.6%→47.1%) — widening the stop makes risk bigger, which makes R:R worse, not better. Investigating why the trade set didn't change at all led to the real discovery below.

**Real finding (Day 95, later same session): the live R:R gate was never the same logic the backtest validated.** Traced `backtest_holistic.py`'s actual Config C entry gate (`check_entry_signals()`) and found it computes R:R from **real support/resistance levels** (`risk = price - nearest_support`, `reward = nearest_resistance - price`, gated on `is_viable AND rr_ratio>=1.2`, falling back to a flat 1.5 estimate only when no S&R levels exist) — completely different from `compute_entry_levels()`'s flat+8%/ATR-clamped-stop formula, which is exit-management logic only in the backtest, never the entry decision. `live_signals.py` has used the exit formula as if it were the entry gate since Day 81 — a live/backtest divergence in the same bug class as Golden Rule 19 (JS/Python verdict parity), just never caught until this session because nobody had compared the two "R:R checks" against each other before. This — not the stop clamp specifically — is the real explanation for the 81% rejection rate measured above.

**Action taken:** Built a parallel **Path B** forward-test experiment (see `PAPER_TRADING_PREREGISTRATION.md` §8b) that runs the real S&R-based gate (`check_sr_gate()` in `live_signals.py`) side-by-side with the unchanged Path A, on the exact same daily candidate pool, tracked under its own `variant='B_revised_rr'` ledger tag with its own 100-trade bar — Path A's frozen count is completely untouched. `paper_trading/ledger.py` gained a `variant` column (migrated cleanly, all existing rows auto-tagged `'A_frozen'`) threaded through every query/write function. `backend.py`'s `/api/paper-trading/status` endpoint was explicitly pinned to `variant='A_frozen'` so the Forward Test tab's UI stays exactly as it's always been — Path B stays a backend/CLI-report-only concept (`daily_job.py --report`) until explicitly surfaced. Verified live: schema migration clean (2 closed/22 open/4 pending momentum, 25 closed/3 open MR, all correctly tagged `A_frozen`, zero cross-contamination), `check_sr_gate()` tested against 6 real tickers (returns sensible real support/resistance levels, no crashes) — though notably even liquid names like NVDA failed the real gate too (R:R 0.05 — near-high momentum stocks often have little room to their *nearest* resistance), consistent with Config C's genuinely low historical trade rate (~15-23/year on 60 tickers), not a new problem. A full daily job run produced 0 new signals for both variants today (consistent with Path A's own recent pace, not itself a red flag). 3-pass review (Golden Rule 32) done: (1) confirmed working live; (2) found and fixed the `/api/paper-trading/status` blending gap; (3) swept every `has_active_or_cooldown()`/`queue_pending_signal()` call site for positional-arg breakage (none found) and confirmed `activate_position()`/`close_position()` don't need changes (variant fixed at insert, untouched by lifecycle transitions).

**Related finding (same investigation):** Checked whether widening momentum's TradingView candidate-pool limit (currently 150, raised 50→150 Day 88) could help instead — it can't, meaningfully. Config C's base screener filter only matches **160 stocks in the entire market today** (checked directly, `limit=500` returned exactly 160 rows) — the current limit=150 already captures ~94% of the whole matching universe. Contrast with MR's universe query, which matches **3,427** stocks against its much looser base filter (only market_cap≥$500M, price>$5) — MR's limit=150 is a deliberate rate-limit-safety cap with huge untapped headroom (Day 89), while momentum's limit=150 is *already* nearly exhausting its universe. Raising momentum's limit to 250-300 would add at most ~10 more daily candidates — not a meaningful lever on its own.
